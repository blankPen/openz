# Openz 移动端功能完善设计文档

**日期**: 2026-06-20
**状态**: 已确认
**分支**: `feature/mobile-complete`

---

## 1. 背景与目标

### 现状
- **Phase 1 完成**: 脚手架、主题系统、原子组件库、BottomSheet
- **Phase 2 未完成**: UI 1:1 复刻设计稿（5 屏）
- **Phase 3 部分完成**: 已接入 daemon 和 relay server，但很多功能仍为 mock 实现

### 目标
在 `feature/mobile-complete` 分支上，通过增量替换 mock 的方式，完善移动端所有核心功能，实现 App 真正可用。

---

## 2. 功能模块优先级

### Phase A - 核心体验（优先实现）
1. **会话管理**
   - 创建新会话
   - 切换会话
   - 删除会话
   - 查看历史会话列表

2. **实时对话**
   - 文字输入发送
   - 消息列表展示
   - AI 响应流式展示

### Phase B - 增强功能
3. **附件处理**
   - 文件上传
   - 文件预览
   - 文件下载

4. **连接诊断**
   - daemon 连接状态
   - relay 连接状态
   - 断线重连机制

### Phase C - 个性化
5. **设置管理**
   - 主题切换（深色/浅色）
   - 模型选择
   - 配置项管理

## 4. 技术路径

### 分支策略
- 新建分支: `feature/mobile-complete`
- 基于当前工作流分支
- 每次 PR 聚焦一个功能模块

### Mock 替换策略
1. 识别现有 mock 实现
2. 定义真实实现的接口
3. 增量替换，逐一验证
4. 保持向后兼容

### 架构微调（如需要）
- 核心架构保持稳定
- 仅在必要时进行小幅调整
- 遵循现有代码模式

---

## 5. 验收标准

- [ ] 所有功能模块从 mock 替换为真实实现
- [ ] UI 完整复刻设计稿
- [ ] TypeScript 类型检查通过
- [ ] 单元测试覆盖新增逻辑
- [ ] App 可在 iOS/Android 真机运行

---

## 6. 实施顺序

1. **会话管理** → 为其他功能提供基础
2. **实时对话** → 核心用户流程
3. **附件处理** → 依赖会话管理
4. **连接诊断** → 辅助功能
5. **设置管理** → 独立模块

---

## 7. 风险与对策

| 风险 | 对策 |
|------|------|
| 现有架构不支持复杂功能 | 微调架构，保持渐进 |
| mock 替换影响现有功能 | 增量替换，充分测试 |
| 设计稿与实现不匹配 | 先验证接口，再实现 |
